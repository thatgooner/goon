## hermes-memory-independent-review

### what is good

**1. frozen prompt snapshot**

Hermes loads memory once per session and freezes the system prompt. writes go to disk immediately but the active prompt stays stable until the next session or compression rebuild. the stored prompt is persisted in SQLite and reused verbatim on session continuation instead of being rebuilt from live files.

why this matters:
- prefix/prompt caching stays valid across the whole session
- behavior does not drift every turn
- prompt rebuild cost is paid once, not per-turn
- session continuation does not accidentally re-inject memory changes the model already wrote

builder verdict: this is the single strongest Hermes pattern. Purr should absolutely have a frozen session pack. the key difference is that Purr's canonical truth must stay live in the ledger — the session pack is a stable view, not the truth itself.

**2. bounded curated memory**

Hermes enforces character limits on memory files. this forces consolidation and prevents prompt bloat. the hot memory is small by design.

why this matters:
- prompt cost stays predictable
- forces quality over quantity
- makes audit easy
- prevents "save everything forever" decay

builder verdict: steal the principle directly. Purr's hot pack should be slot-capped and token-budgeted. the larger structured ledger can hold much more, but what enters the prompt must be bounded.

**3. transcript/session search layer**

raw transcript lives in SQLite with FTS5. the agent can search past sessions lexically and then summarize only the matched ones cheaply. long-tail history stays out of the hot prompt.

why this matters:
- long-term recall without prompt stuffing
- cheap search-then-summarize is cost-effective
- raw history stays durable and searchable

builder verdict: the philosophy is right. Purr should also separate raw history from hot memory. but Purr's recall needs to go beyond lexical — structured filters first, semantic support second, transcript evidence third.

**4. memory nudge + pre-compression flush behavior**

Hermes intends to nudge the model after enough turns to save memory, and runs a dedicated pre-compression salvage step where only the memory tool is available.

critical code-grounded finding from ily/07: the memory nudge loop is effectively dead. `_turns_since_memory` resets at the start of every turn before it can accumulate. the pre-compression salvage step is real and working — it injects a hidden save-everything turn before compression, then strips the artifacts from history.

why this matters:
- the salvage step is genuinely good operational memory capture
- the nudge being broken shows why chat-loop heuristics are fragile
- Purr cannot rely on model-initiative for persistence

builder verdict: steal the salvage step concept. extend it to session-end, long-idle cutoff, archive/handoff, and mobile re-entry boundaries. reject chat-loop nudges as a primary persistence mechanism — use event-driven server-side extraction instead.

**5. session-lineage continuation after compression**

when Hermes compresses, it does not overwrite the old session. it closes the old session, creates a new child session with `parent_session_id`, and links them. titles get auto-numbered (`foo`, `foo #2`). session_search can traverse child→root.

why this matters:
- audit trail survives compression
- pre-compression raw sessions remain searchable
- summarization does not become the only truth

weak part: title numbering is UX convenience, not a real lineage graph. child-session search resolves back to the root parent, which can blur the exact matched evidence.

builder verdict: steal the parent-child lineage idea. reject title-based pseudo-lineage. Purr needs explicit `session_window` and `episode` objects with real ids and parent links.

**6. compression structural integrity**

Hermes avoids splitting tool-call/result groups at compression boundaries, removes orphan tool results, and inserts stub results when needed.

builder verdict: any Purr background-cognition or archive pipeline must guard message-graph integrity the same way. summaries are not enough — causal/evidence structure matters.

**7. memory security scanning**

Hermes scans tool-mediated memory writes for prompt injection and exfiltration patterns.

gap: load-time re-injection of existing memory files is not scanned. manual or external writes can bypass write-time scanning.

builder verdict: Purr must sanitize on ingest, on promotion to hot-pack material, and on any network/social-derived content. treat "future prompt material" as a security boundary every time.

### why it is good (summary thesis)

Hermes is not good because it has memory files. it is good because it treats memory as a **controlled behavior pipeline**: freeze the prompt, separate raw history from curated memory, recall on demand instead of stuffing, salvage before compression, protect structural integrity.

the discipline is the real asset, not the implementation.

---

### what breaks for purr

**limitation 1: flat text memory with no lifecycle**

Hermes memory entries are delimiter-separated text blocks. no id, no kind, no state, no confidence, no salience, no evidence refs, no supersedes/conflicts graph, no review schedule. mutation is substring-based replace/remove.

for purr: contradiction handling becomes text surgery instead of state transitions. as memory grows, ambiguous substring matches become normal. there is no way to track identity drift, episodic evidence, pattern confidence, or staleness.

**limitation 2: same-session stale truth**

the frozen snapshot is great for cost control but means same-session corrections are invisible to the active prompt. user says something, Purr writes it to memory, but the reply still reasons from the old snapshot.

for purr: this breaks the "i know you" fantasy faster than it breaks a normal assistant. if the user corrects something and the next reply ignores it, the product feels fake. Purr needs a frozen session pack plus a live turn-level override lane for direct corrections, fresh contradictions, and high-value new facts.

**limitation 3: recall is not user-scoped**

`search_messages()` filters by source/platform, not by a per-user memory boundary. child-session hits can get resolved to the root parent, potentially missing the exact matched evidence. the current-session exclusion is ID-local, not lineage-aware.

for purr: `1 human = 1 purr` means every retrieval path must start with `owner_id` + `purr_id`. cross-user bleed is a product-killer. evidence-safe recall must preserve exact hit segments before lineage recaps.

**limitation 4: heuristic-dependent memory capture**

the nudge loop is dead in practice. pre-compression salvage works, but outside that moment, persistence depends on the model choosing to call the memory tool. if the model does not call it, valuable context simply vanishes.

for purr: memory extraction must be event-driven and deterministic-ish. server-side classifiers/extractors should identify candidates independently. save failures must be visible and recoverable.

**limitation 5: lexical recall ceiling**

FTS5 is useful but cannot handle paraphrase, fuzzy emotional patterns, latent habit clusters, or relational memory queries like "this is the same avoidance pattern again."

for purr: hybrid recall is required. structured filters lead (kind, state, confidence, owner scope), semantic support assists, transcript evidence anchors.

### what to reuse

steal directly:
- frozen session pack (but not frozen truth)
- bounded hot-pack discipline
- raw transcript store separated from curated memory
- pre-compression salvage step (extend to more boundaries)
- stored prompt snapshot reuse across session continuation
- structural integrity guards during compression
- security scanning mindset for re-injected content
- search → summarize cost discipline

### what to avoid

do not copy:
- flat MEMORY.md / USER.md as the product memory model
- substring-based mutation as the main update path
- same-session stale truth with no live override lane
- unscoped recall (no per-user boundary)
- title-based pseudo-lineage standing in for real episode graph
- chat-loop nudges as the primary memory capture mechanism
- one global memory namespace
- no confidence / no review / no staleness tracking

## first-run checklist answers

### 1. Hermes'ten neyi calmaliyiz?

the behavior discipline, not the implementation.

specifically:
- frozen session pack pattern (stable prompt, once per session)
- bounded hot-memory budget (forces quality)
- transcript store separate from curated memory
- pre-compression salvage step
- stored prompt reuse on session continuation
- compression with lineage (parent-child sessions)
- structural integrity during compression
- security scanning for persistent prompt material

### 2. Hermes'ten neyi almamaliyiz?

- flat text as the source of truth
- substring mutation
- stale same-session reads with no correction lane
- lexical-only recall
- global unscoped memory namespace
- model-initiative-dependent persistence
- summaries as replacement for canonical truth

### 3. Purr'u builder gozuyle bir cumlede nasil tarif edersin?

Purr is a memory-first 1:1 creature that needs a structured per-user ledger, bounded retrieval packing, a live correction override lane, and event-driven background cognition — all invisible behind a single chat surface.

### 4. first implementation slice ne olmali ve neden?

**memory-ledger** should be first.

why:
- every other component depends on structured memory items existing
- retrieval packer needs something to pack from
- correction loop needs state to transition
- salvage/extraction needs a target store
- without the ledger, everything else is either building on flat text or building on nothing

the ledger gives the whole system its truth backbone. it does not need to be fancy — Supabase table(s) with typed rows, lifecycle state, owner scoping, and evidence refs. but it needs to exist before anything else ships.

second slice should be retrieval/context-packer (the bridge between ledger and prompt).
third slice should be the correction/review loop (makes the ledger actually useful over time).
