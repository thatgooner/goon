## session-scope-and-lineage-contract-note

### owner/purr scoping

Identity scope must be the first filter on every durable row and every retrieval path.

Explicit rejections:
- **platform-global scope** — never scope by gateway/platform/source alone
- **title-based scope** — titles are UX convenience, never keys, never retrieval scope
- **source-only filters** — identity boundary before relevance/recency/semantics

Canonical keys (mandatory on every row): owner_id, purr_id, memory_lane (private_1_1 | public_safe | catnet | system_ops).

Filter order: owner_id → purr_id → memory_lane → episode/window eligibility → then ranking/search.

### session_window vs episode

| concept | scope | purpose |
|---------|-------|---------|
| session_window | active pack, live reply, compression boundary | prompt-driving state, pack versioning, re-entry artifact |
| episode | historical grouping, relationship chapters | retrieval grouping, lineage context, recap structure |

session_window fields: window_id, owner_id, purr_id, episode_id, parent_window_id, entry_surface, window_state, opened_at, closed_at, stored_session_pack_id, stored_reentry_pack_id.

episode fields: episode_id, owner_id, purr_id, kind, started_at, ended_at, status, summary_ref.

### resume/reentry continuation policy

- compression creates child window, not rewritten past
- re-entry creates new window (warm return → same episode; long idle → new episode)
- server owns continuity; client only resumes it
- client resets must not redefine ownership or scope

### exact-hit retrieval vs lineage recap

- preserve exact message_id/span, window_id, episode_id first
- lineage context is enrichment, not replacement for exact hit
- lineage walk only when exact-window hits are weak or summary is requested
- current-window exclusion must be lineage-aware

### pack artifact ownership

| artifact | scoped to |
|----------|-----------|
| session_pack | window_id |
| turn_delta_pack | window_id |
| reentry_pack | boundary between closed and continuation window |
| proactive_pack | purr_id (references target window/episode) |

## phase-one-build-order

### first slice: memory-ledger

Schema, persistence, and read/write API for the memory system of record. Implements identity scope (owner_id + purr_id), session_window, episode lineage, and lifecycle states. SQL migrations, typed contracts, minimal API.

Why first: every other component depends on it. extractor writes into it, packer reads from it, feedback-orchestrator corrects within it. without the ledger, everything else builds on nothing.

### second slice: memory-candidate-extractor

Turn-critical tool that extracts memory candidates from conversation turns and writes them into the ledger. Implements intake rules (ask_now / defer / silent_store / drop).

Why second: first producer in the turn-critical lane. once the ledger exists, the extractor is the first consumer that fills it.

Depends on: memory-ledger.

### third slice: memory-context-packer

Reads from ledger, produces 4 pack artifacts (session_pack, turn_delta_pack, reentry_pack, proactive_pack). Slot-based caps, trust suppression, pack order, patch vs rebuild.

Why third: bridge between ledger and model prompt. without it, memories exist but cannot selectively reach the response.

Depends on: memory-ledger + memory-candidate-extractor (so there is content to pack).
