## prediction-memory-and-proactive-timing-note

### prediction memory kinds

Prediction is a backend memory capability, not a user-facing feature. It improves retrieval, feedback timing, roast/nudge decisions, and proactive ping quality.

**`pattern_signal`** — recurring rhythms and transitions.
Fields: memory_id, owner_id, purr_id, kind=pattern_signal, state, confidence, salience, pattern_type, trigger_context, horizon (turn|session|daily|long), evidence_refs, last_hit_at, needs_review_at, expires_at, conflicts_with, supersedes.
Examples: late-night stress → topic loop; after joke X → insecurity Y; 3-day silence → apology re-entry.

**`open_loop`** — unresolved threads with future pull.
Fields: memory_id, owner_id, purr_id, kind=open_loop, state, confidence, salience, loop_type, opened_at, last_referenced_at, needs_review_at, resolved_by, evidence_refs.
Examples: promised to send something; unfinished confession; avoided topic likely to return.

**`next_action_candidate`** — short-lived operational predictions.
Fields: memory_id, owner_id, purr_id, kind=next_action_candidate, state, confidence, salience, predicted_action_type, horizon, trigger_context, why_now, evidence_refs, expires_at, last_hit_at, last_miss_at.
Examples: likely next ask, follow-up topic, mood lane, proactive ping opportunity. Decay quickly unless they keep hitting.

### horizons

| horizon | scope | rules |
|---------|-------|-------|
| turn | next 1-3 msgs | highest recency, shortest TTL, pack only if it changes the reply |
| session | this chat window | medium TTL, combine with open-loop weight |
| daily | later today / re-entry | notification timing, cooldown-aware, mostly server-side |
| long | weeks+ | promote only with repeated evidence, slower decay, periodic review |

### retrieval budget

Rank order: corrections → preferences → open loops → pattern signals → 1-2 next-action candidates → episodic evidence → transcript fallback.

Hot pack: 0-2 prediction hints max. If a prediction needs paragraphs to justify itself, it stays out of the pack.

### background jobs

**Lane A (turn-critical):** message_intake_extractor, direct_correction_detector, live_override_injector, retrieval_packer
**Lane B (deferred):** memory_consolidation_worker, contradiction_resolver, session_salvage_worker, sanitization_guard
**Lane C (heartbeat):** review_scheduler, review_executor, proactive_timing_scorer, proactive_message_planner

### failure modes

1. surveillance slop — low-confidence predictions stay internal; never state speculation as fact
2. stale correction — direct_correction_detector + live_override_injector
3. pattern landfill — TTL, decay, evidence thresholds, aggressive dropping
4. summary becomes truth — summaries are navigation only; canonical truth = ledger + evidence_refs
5. notification creepiness — cooldown + confidence; notification text less revealing than internal rationale
6. vector brainrot — structured filters first; vector only as support

## memory-lifecycle-and-feedback-state-machine-note

### canonical states

| state | definition |
|-------|-----------|
| candidate | enough evidence to keep, not enough trust for durable truth |
| confirmed | trusted enough to influence behavior without constant re-checking |
| stale | used to be trusted, now needs caution (not rejection) |
| rejected | not valid truth; kept for audit and anti-relearn |
| superseded | was once valid, replaced by newer truth |

Overlay states: review_status (none|queued|due|ask_now|snoozed), contradiction_status (clean|challenged), pack_policy (hot|shadow|suppress|never).

Transitions: candidate→confirmed (explicit confirmation, direct statement, repeated evidence); candidate→rejected (user denies, evidence breaks it); confirmed→stale (volatility expiry, repeated misses); stale→confirmed (reconfirmation); confirmed→superseded (new truth replaces old).

### ask_now / defer / silent_store / drop rules

Scoring factors: leverage, explicitness, contradiction_pressure, volatility, sensitivity, timing_fitness.

| action | when | effect |
|--------|------|--------|
| ask_now | direct correction, contradiction blocks reply, high-leverage, boundary/safety | create/update immediately, mark old challenged, inject live_override |
| defer | valuable but not urgent, ambiguous | store as candidate, review_status=queued, needs_review_at |
| silent_store | explicit, low-risk, low-ambiguity, not creepy | candidate, pack_policy=shadow, light retrieval influence |
| drop | low evidence, low leverage, transient, duplicate, too creepy | no durable row or ephemeral extraction logs only |

Same-session corrections: if waiting makes the next reply feel fake, it goes into live_override.

### contradiction handling

When new evidence conflicts with confirmed memory:
- old item → contradiction_status=challenged, pack_policy=suppress
- new item → enters as candidate with high contradiction pressure

Resolution: explicit correction → old superseded, new confirmed. Ambiguity → both out of factual pack, review queued. Joke/exception → old returns to clean, new episode-only or rejected.

Hard rule: never pack two incompatible truths as equal active facts.

### review cadence

| kind | cadence |
|------|---------|
| profile/stable facts | 90-180 days |
| preferences | 14-45 days |
| relationship memory | 30-60 days |
| open loops | hours to 7 days |
| pattern signals | hours to 60 days |
| next-action candidates | short TTL, validate internally |

Anti-spam caps: max 1 explicit check per session, max 1 proactive review ping per day, max 3 attempts per memory before heavy cooldown.

### retrieval / proactive gating

Pack policy: hot (in pack), shadow (soft influence), suppress (out until resolved), never (rejected/superseded).

Suppression: challenged → suppress from factual pack. stale → only if directly relevant, lower weight. candidate → never dominate. rejected/superseded → never.

Proactive gates: only from confirmed open loops, repeated patterns, recent hit-rate. Never from candidates, challenged, stale without reconfirmation. contradiction_status=challenged reduces proactive score.

## retrieval-context-packer-and-pack-lifecycle-note

### pack artifacts

| artifact | purpose | budget target | hard cap |
|----------|---------|---------------|----------|
| session_pack | stable memory for current conversation window | 500-800 tokens | 1000 tokens |
| turn_delta_pack | tiny live patch for same-session freshness | 30-120 tokens | 180 tokens |
| reentry_pack | compact resume for mobile/webview re-entry | 350-650 tokens | 800 tokens |
| proactive_pack | internal-only for first-text decisions (stricter) | 120-250 tokens | 350 tokens |

### slot caps (session_pack)

Lane A — non-negotiable truth: hard constraints max 3, identity/profile max 3, active preferences max 4.
Lane B — relationship state: texture max 3, open loops max 3.
Lane C — situational: episodic anchors max 2, pattern signals max 2, next-action candidates max 0-2.
Lane D — evidence: micro-excerpts max 2, transcript snippets only as last resort.

Pack order: turn_delta first → hard truths → preferences → relationship → open loops → patterns → episodic → 0-2 prediction hints → evidence → transcript last.

Trust suppression: challenged → never factual slots; candidate → shadow only; stale → only if relevant, lower weight; rejected/superseded → never.

### budget envelopes

Budget cuts when tight (in order): transcript fallback → extra evidence → weak episodic → second prediction hint → low-salience patterns → low-impact texture → LAST: core boundaries, preferences, open loops.

Free-tier rule: cut throughput, model strength, proactive frequency BEFORE core memory integrity. Memory quality is the product.

### patch vs rebuild triggers

| behavior | trigger | effect |
|----------|---------|--------|
| reuse (default) | normal turns, low-value candidate stored, weak pattern changed | use stored session_pack as-is |
| patch | explicit correction, boundary change, high-leverage preference, contradiction | inject turn_delta_pack, suppress challenged truth, let next rebuild absorb |
| partial rebuild | new important open loop, loop resolves, 3+ deltas stacked, 2+ lanes changed | queue after reply, not inline |
| full rebuild | re-entry after idle, compression/archive, pack corruption, multiple contradictions on core identity | rebuild from ledger + latest episode anchors |

Idle rule: same active window → reuse. Same day warm return → reentry_pack. Long idle or major drift → full rebuild.

### mobile re-entry behavior

Pack artifacts live server-side. Webview is thin client for display and input only. pack_cache_builder precomputes session_pack and reentry_pack after turns and before expected re-entry windows.

Cold start: user taps notification → system loads cached reentry_pack from server, not webview memory. Re-entry is a server event. Warm return creates new child window in same episode. Long idle or topic break creates new episode + new window.

Good: user comes back and purr is still "there." Bad: every re-entry feels amnesiac because no stable pack artifact existed.
